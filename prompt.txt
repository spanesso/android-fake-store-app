0. REGLAS INVIOLABLES (leer antes de ejecutar cualquier acción)
────────────────────────────────────────────────────────────────────────────────

R0.1  IDIOMA — Todos los documentos, comentarios de alto nivel, mensajes de
      commit, descripciones de PR, especificaciones, planes, tareas, READMEs y
      documentación técnica se escriben en ESPAÑOL. El código fuente (nombres
      de clases, funciones, variables) sigue convenciones de Kotlin/Android en
      inglés.

R0.2  ARQUITECTURA — Clean Architecture estricta en TODOS los módulos de
      feature, con tres capas separadas en submódulos o paquetes:
        · data    → fuentes de datos (red, BD), DTOs, mappers, repositorios.
        · domain  → entidades, casos de uso (interactors), contratos
                    (interfaces de repositorio).
        · presentation → MVVM (ViewModel + UiState + UiEvent + UiEffect).
      Las dependencias APUNTAN HACIA EL DOMINIO. data y presentation NUNCA se
      conocen entre sí. La comunicación entre capas y módulos SIEMPRE va por
      INTERFACES (contratos). Esto es no negociable: facilita testeo, mocking
      y reemplazo de implementaciones.

R0.3  VISTAS SIN VIEWMODEL — Los Composables que renderizan UI (carpeta
      `ui/components` o `ui/screens`) NUNCA reciben ni instancian un
      ViewModel. Reciben:
        · Un `UiState` (data class inmutable) como parámetro.
        · Lambdas `(Event) -> Unit` para notificar acciones del usuario.
      Existe SIEMPRE un Composable wrapper "Route" o "Container" en una capa
      superior que sí conoce el ViewModel y conecta el state/eventos al
      Composable puro. Esto permite previews completos y reutilización total.

R0.4  PREVIEWS OBLIGATORIOS — Cada Composable de vista DEBE tener `@Preview`
      para CADA estado de UI posible (Loading, Error, Empty, Content,
      Content+favoritos, etc.) y en variantes claro/oscuro.

R0.5  TESTS POR MÓDULO — Cada módulo creado lleva sus pruebas unitarias en el
      mismo PR. Cobertura mínima 80% en domain y data; presentation 70%;
      design-system con snapshot tests.

R0.6  DESIGN SYSTEM PRIMERO — Ningún componente visual (botón, input, label,
      card, badge, dialog, etc.) se usa directamente desde Material3 en las
      pantallas de feature. Todo pasa por wrappers del módulo `:core:design-system`
      configurables vía parámetros.

R0.7  SECRETS FUERA DEL CÓDIGO — Ninguna API key, endpoint privado, token de
      Amplify o credencial vive en el repositorio. Se usan `local.properties`
      + `BuildConfig`, NDK para strings críticos, y para producción se
      inyectan vía CI/CD desde un vault. Documentar la estrategia en
      `docs/seguridad.md`.

R0.8  COMMITS — Conventional Commits en español: `feat:`, `fix:`, `docs:`,
      `test:`, `refactor:`, `chore:`, `ci:`. Mensajes claros, scope por
      módulo (`feat(products): añade caso de uso ObtenerProductos`).

R0.9  WORKFLOW — Usar Spec Kit. Para cada feature/módulo nuevo seguir el
      ciclo: `/speckit-specify` → `/speckit-clarify` → `/speckit-plan` →
      `/speckit-tasks` → `/speckit-analyze` → `/speckit-implement`. Los
      artefactos generados (`spec.md`, `plan.md`, `tasks.md`) están en
      español.

R0.10 NO IMPROVISAR — Antes de codificar una pantalla o módulo, leer su
      `spec.md` y `plan.md`. Si algo no está claro, ejecutar
      `/speckit-clarify`, no asumir.

R0.11 ERRORES TIPADOS — Ninguna capa propaga `Throwable` o `Exception` cruda
      hacia arriba. Toda salida de un repositorio o caso de uso devuelve
      `Either<DomainError, T>` (o equivalente). Las excepciones se capturan
      en la frontera data y se traducen a `DomainError`. La UI nunca muestra
      `throwable.message`: muestra mensajes vía `stringResource(...)` desde

R0.12 SOLO HERRAMIENTAS CON TIER GRATUITO — No se incorpora al proyecto

        · CI/CD: GitHub Actions (2000 min/mes en repos privados, ilimitado

        · Distribución interna: Firebase App Distribution (gratis).
        · Pruebas en dispositivo: Firebase Test Lab (cuota gratis diaria).
        · Calidad: Detekt + ktlint + Kover (locales, gratis). SonarQube

      Si una sección de este documento menciona una herramienta de pago,

R0.13 SKILL-CREATOR — Los skills definidos en §13 se generan UTILIZANDO el

────────────────────────────────────────────────────────────────────────────────
3. ARQUITECTURA — CLEAN + MVVM
────────────────────────────────────────────────────────────────────────────────

Capa Domain (pura Kotlin, sin Android)
  · Entidades: data classes inmutables, sin anotaciones de framework.
  · UseCases: una clase con `operator fun invoke(...)`, una responsabilidad,
    devuelve `Either<DomainError, T>`.
  · Repository interfaces: contratos puros, definidos aquí, implementados en
    data; firmas devuelven `Either<DomainError, T>` o `Flow<Either<...>>`.
  · Errors tipados: `sealed class DomainError` (ver §7.2).
  · Result wrapper: `Either<DomainError, T>` (Arrow). Decisión registrada
    en `docs/adr/0001-manejo-errores.md`.

Capa Data
  · Repositorios: implementan contratos de domain. Coordinan fuentes (remote
    + local). Capturan excepciones en su frontera y mapean a `DomainError`
    con un `ErrorMapper` testeado.
  · Remote: Retrofit services + DTOs + Mappers (DTO→Domain).
  · Local: Room DAOs + Entities + Mappers (Entity↔Domain).
  · Single Source of Truth: para favoritos y productos cacheados → Room.
    NetworkBoundResource pattern o equivalente con Flow.
  · Mappers explícitos: NUNCA exponer DTOs ni Entities fuera de data.

Capa Presentation (MVVM)
  · ViewModel:
      - Expone `val uiState: StateFlow<UiState>`.
      - Expone `val uiEffect: SharedFlow<UiEffect>` para efectos one-shot
        (snackbar, navegación).
      - Recibe `fun onEvent(event: UiEvent)`.
      - NUNCA referencia tipos Compose.
      - Recibe casos de uso por constructor (Hilt).
      - Tiene `CoroutineExceptionHandler` raíz que reporta a telemetría
        cualquier excepción no esperada y emite `UiState.Error` o
        `UiEffect.ShowError`.
  · UiState: data class inmutable con sub-estados (loading, error, content).
    Preferir sealed para el modo:
        sealed interface UiState {
          data object Loading : UiState
          data class Error(val error: UiError) : UiState
          data class Content(val items: List<ProductoUi>) : UiState
        }
  · UiEvent: sealed interface con acciones del usuario (incluye `Retry`).
  · UiEffect: sealed interface con efectos one-shot (incluye
    `ShowSnackbar(uiError)`).
  · Composable Route (con ViewModel) → Composable Screen (puro) → Composables
    granulares del design system.

Comunicación entre módulos
  · Solo a través de interfaces expuestas en módulos `:*:api`.
  · Las implementaciones viven en módulos `:*:impl` y se inyectan con Hilt
    `@Binds`.

6. SEGURIDAD
────────────────────────────────────────────────────────────────────────────────

6.1 Anti-ingeniería inversa
    · R8 con reglas agresivas: `-allowaccessmodification`,
      `-repackageclasses`, `-overloadaggressively`.
    · ProGuard rules específicas por módulo en `consumer-rules.pro`.
    · Ofuscación de strings sensibles: mover a NDK (C/C++) o usar
      `StringFog`/`Paranoid` plugin.
    · DEX shrinking en release.
    · Documentar lo que NO se ofusca (Compose, Hilt, Retrofit interfaces)
      y por qué.

6.2 Detección de root/jailbreak
    · Integrar RootBeer + checks propios (build tags, magisk, su binary).
    · Detección de emulador en producción.
    · Política configurable: bloquear, advertir, o solo loggear.
    · Implementar en `:core:security` con interfaz `IntegrityChecker`.

6.3 Anti-screenshot
    · `FLAG_SECURE` configurable por pantalla vía un Composable
      `SecureScreen { content }` que aplica el flag al window de la
      Activity.
    · Aplicar a Profile y a cualquier pantalla con datos sensibles.

6.4 Autenticación biométrica fuerte
    · BiometricPrompt con `BIOMETRIC_STRONG`.
    · Fallback a PIN/contraseña del dispositivo solo si está disponible.
    · Re-autenticación tras tiempo de inactividad configurable.
    · Vinculación a Keystore: las claves criptográficas que protegen tokens
      requieren autenticación biométrica reciente.

6.5 Protección de servicios web
    · Certificate pinning con OkHttp `CertificatePinner` (pines primario +
      backup).
    · `network_security_config.xml` con dominios permitidos y pins.
    · No claves de Amplify en código: usar AWS Cognito Identity Pool +
      roles IAM o un BFF (Backend for Frontend) que firme requests; si no
      es posible, mover claves al backend o usar tokens efímeros.
    · TLS 1.2+ mínimo.
    · OkHttp interceptor que firma requests con HMAC (si aplica).

6.6 Almacenamiento seguro
    · Room cifrado con SQLCipher.
    · EncryptedSharedPreferences / EncryptedDataStore para tokens.
    · Android Keystore para claves de cifrado.
    · No PII en logs en release.

6.7 Hardening adicional
    · Validar firma APK en runtime (anti-tampering).
    · Detección de debugger en release (`Debug.isDebuggerConnected`).
    · Hooks anti-Frida/Xposed (best effort).
    · Documentar todo en `docs/seguridad.md` con threat model.

────────────────────────────────────────────────────────────────────────────────
7. MANEJO DE ERRORES (estrategia profesional, transversal)
────────────────────────────────────────────────────────────────────────────────

7.1 Principios
    · Errores explícitos y tipados — nunca strings ni excepciones genéricas
      filtradas a la UI.
    · Cada capa habla SU propio idioma de errores; los traduce al pasar a
      la siguiente capa.
    · Fail-fast en bugs (precondiciones con `require`/`check`); fail-soft
      en errores esperados (red, validación, BD).
    · Telemetría obligatoria: todo error mapeado se loguea con contexto
      (module, useCase, code, sessionId), SIN PII.
    · Los mensajes para el usuario son cortos, claros, accionables y
      localizados.

7.2 Modelo de DomainError (en `:core:error` y/o `:domain` por feature)

    sealed interface DomainError {
      val cause: Throwable?
      sealed interface Network : DomainError {
        data class NoConnection(override val cause: Throwable? = null) : Network
        data class Timeout(override val cause: Throwable? = null) : Network
        data class Server(val httpCode: Int, override val cause: Throwable? = null) : Network
        data class Unauthorized(override val cause: Throwable? = null) : Network
        data class Forbidden(override val cause: Throwable? = null) : Network
        data class NotFound(override val cause: Throwable? = null) : Network
        data class Parsing(override val cause: Throwable? = null) : Network
      }
      sealed interface Database : DomainError {
        data class ReadFailed(override val cause: Throwable? = null) : Database
        data class WriteFailed(override val cause: Throwable? = null) : Database
        data class NotFound(override val cause: Throwable? = null) : Database
        data class IntegrityViolation(override val cause: Throwable? = null) : Database
      }
      sealed interface Security : DomainError {
        data object BiometricUnavailable : Security { override val cause = null }
        data object BiometricLockout    : Security { override val cause = null }
        data object RootDetected        : Security { override val cause = null }
        data object IntegrityFailed     : Security { override val cause = null }
        data object SessionExpired      : Security { override val cause = null }
      }
      data class Validation(val fields: Map<String, String>) : DomainError {
        override val cause: Throwable? = null
      }
      data class Unknown(override val cause: Throwable? = null) : DomainError
    }

7.3 Wrapper de resultado
    · Usar Arrow `Either<DomainError, T>` en domain (preferido) o un
      `sealed class DomainResult<out T>` propio si se evita Arrow.
    · Composición con `fold`, `map`, `flatMap` — prohibido `when` exhaustivo
      anidado salvo en la frontera de presentación.
    · Para flujos: `Flow<Either<DomainError, T>>`.
    · Decisión registrada en `docs/adr/0001-manejo-errores.md`.

7.4 Mapeo de excepciones (frontera data)
    · Centralizado en `NetworkErrorMapper` y `DatabaseErrorMapper`.
    · Reglas mínimas:
        IOException, UnknownHostException        → Network.NoConnection
        SocketTimeoutException, TimeoutException → Network.Timeout
        HttpException(401)                       → Network.Unauthorized
        HttpException(403)                       → Network.Forbidden
        HttpException(404)                       → Network.NotFound
        HttpException(in 500..599)               → Network.Server(code)
        SerializationException                   → Network.Parsing
        SQLiteConstraintException                → Database.IntegrityViolation
        SQLiteException                          → Database.ReadFailed/WriteFailed
        cualquier otra                           → Unknown(cause)
    · Cada repositorio envuelve sus llamadas con un helper único:
        suspend fun <T> safeApiCall(block: suspend () -> T): Either<DomainError, T>
        suspend fun <T> safeDbCall(block: suspend () -> T): Either<DomainError, T>
    · Tests de mapper por cada rama.

7.5 UiError (capa de presentación)
    · `UiError` es una data class diseñada para UI:

        data class UiError(
          @StringRes val messageRes: Int,
          val severity: Severity,           // Info, Warning, Blocking, Fatal
          val actions: List<UiErrorAction>, // Retry, Dismiss, Login, OpenSettings
          val errorCode: String,            // p.ej. "NET-401", para soporte
        ) {
          enum class Severity { Info, Warning, Blocking, Fatal }
          sealed interface UiErrorAction {
            data object Retry : UiErrorAction
            data object Dismiss : UiErrorAction
            data object Login : UiErrorAction
            data object OpenSettings : UiErrorAction
          }
        }

    · `DomainErrorToUiErrorMapper` convierte cada `DomainError` en `UiError`
      con `messageRes` localizable (`R.string.error_red_sin_conexion`,
      `R.string.error_servidor`, etc.).
    · La UI NUNCA recibe un `DomainError` ni un `Throwable`.

7.6 Estados de error en UI
    · `UiState.Error(uiError)` para fallos bloqueantes de pantalla → render
      `MangoErrorState(uiError, onRetry)`.
    · `UiEffect.ShowSnackbar(uiError)` para errores no bloqueantes →
      `MangoSnackbar` con severidad y acción.
    · `MangoOfflineBanner` global cuando no hay red (escucha
      `ConnectivityObserver` de `:core:network`).
    · Distinguir SIEMPRE Empty de Error.
    · Botón Retry visible y accionable en cualquier `UiState.Error`
      recuperable.

7.7 Reintentos y resiliencia (data)
    · OkHttp `Interceptor` con backoff exponencial para errores transitorios
      (timeouts, 5xx) — máx. 3 intentos, jitter incluido.
    · No reintentar 4xx (excepto 408, 429 con `Retry-After`).
    · Circuit breaker simple para endpoints repetidamente caídos.
    · Operaciones idempotentes solamente; las no idempotentes (POST sin
      idempotency-key) NO se reintentan.

7.8 Coroutines y excepciones
    · `CoroutineExceptionHandler` raíz en cada ViewModel — reporta a
      telemetría y emite `UiState.Error(Unknown)`.
    · `SupervisorJob` en `viewModelScope` para que un hijo fallido no
      cancele hermanos.
    · Cancelación cooperativa: `ensureActive()` antes de operaciones
      costosas.
    · NUNCA `try/catch` genérico en domain — solo en frontera data y en
      handlers raíz.

7.9 Errores de seguridad y autenticación
    · 401 / `SessionExpired` → invalidar sesión, lanzar `UiEffect.NavigateTo
      (LoginBiometrico)`.
    · 403 → mensaje genérico ("No tienes permiso"), no revelar existencia
      del recurso.
    · `BiometricLockout` → bloqueo con cuenta atrás; tras N fallos exigir
      credencial del dispositivo.
    · `RootDetected` / `IntegrityFailed` → política configurable
      (block / warn / log) decidida en `IntegrityPolicy`.

7.10 Reportería y observabilidad de errores
     · Todo `DomainError` (excepto `Validation` y `BiometricLockout`
       esperados) se reporta como NON-FATAL a Firebase Crashlytics.
       Datadog Logs es destino OPCIONAL solo si está disponible (R0.12).
     · Tags estándar (Crashlytics custom keys): `module`, `useCase`,
       `errorType`, `errorCode`, `httpCode?`, `sessionId`, `flavor`,
       `appVersion`.
     · Rate-limit por `errorCode` para no saturar telemetría (máx. N por
       minuto).
     · Errores fatales (uncaught) → Crashlytics con symbolication subida en
       el job de release.
     · NUNCA loguear: tokens, contraseñas, datos biométricos, PII de perfil.

7.11 Internacionalización de mensajes
     · Todo mensaje al usuario vive en `res/values/strings.xml` y
       `res/values-en/strings.xml`.
     · Convención: `error_<dominio>_<causa>` (ej.
       `error_red_sin_conexion`, `error_red_servidor`,
       `error_bd_lectura`).
     · Plurales con `<plurals>` y formatos por locale.

7.12 Validación de datos (entrada del usuario)
     · Validadores en domain: `interface Validator<T> { fun validate(value: T): Either<DomainError.Validation, T> }`.
     · Composición con `traverse` para validar formularios completos.
     · La UI muestra el mapa `fields` resaltando inputs y mensajes
       localizados; el botón submit se deshabilita si hay errores.

7.13 Reglas Detekt específicas para errores
     · Prohibir `catch (e: Exception)` y `catch (e: Throwable)` salvo en
       barreras documentadas (mappers, CoroutineExceptionHandler raíz, init
       de SDK) — anotar con `@SuppressWithReason`.
     · Prohibir `runCatching` en repositorios y usecases sin un `.fold` que
       devuelva `Either<DomainError, T>`.
     · Prohibir lanzar `RuntimeException`/`IllegalStateException` desde
       domain salvo precondiciones (`require`/`check`).
     · Prohibir uso de `throwable.message` en Composables.
     · Prohibir strings hardcoded en `MangoErrorState`, `MangoSnackbar`,
       diálogos — debe pasarse por `stringResource`.

7.14 Tests de errores (obligatorio por módulo)
     · UseCase: happy path + UN test por cada `DomainError` que pueda
       devolver.
     · Repository: con MockWebServer simular 200, 4xx, 5xx, JSON inválido,
       timeout, sin red. Con InMemoryRoom simular constraint violation.
     · ViewModel con Turbine: verificar transición Loading → Error
       (con UiError correcto) y que `Retry` produce Loading → Content/Error.
     · Snapshot tests del `MangoErrorState`, `MangoSnackbar` y
       `MangoOfflineBanner` para cada severidad.

7.15 Skill de auditoría
     · El skill `validar-manejo-errores` (§13) audita todo lo anterior y
       reporta violaciones por archivo:línea.

13. SKILLS A CREAR ANTES DE PROGRAMAR (PRIORIDAD MÁXIMA)
────────────────────────────────────────────────────────────────────────────────

Ubicación: `.claude/skills/<nombre-skill>/SKILL.md`
Cada skill incluye descripción YAML frontmatter y un cuerpo con
instrucciones detalladas + plantillas. Todos los outputs de los skills
están en español.

PROCEDIMIENTO OBLIGATORIO DE CREACIÓN (R0.13)
  Cada uno de los 7 skills se genera invocando el skill
  `example-skills:skill-creator` (plugin instalado).

  Por cada skill a crear:
    1. Invocar `example-skills:skill-creator` con un brief que incluya:
         - Nombre exacto del skill (kebab-case, en español).
         - Descripción una-línea (qué hace, cuándo dispararlo).
         - Triggers / cuándo Claude debe invocarlo.
         - Inputs esperados.
         - Outputs esperados (archivos, ubicación).
         - Restricciones de idioma (siempre español) y de arquitectura
           (referenciar §0, §3, §7 cuando aplique).
         - Plantillas que el skill debe contener internamente.
    2. Revisar el `SKILL.md` generado: validar frontmatter (`name`,
       `description`), longitud y especificidad de la descripción
       (debe disparar correctamente sin falsos positivos).
    3. Si el skill creado no cumple los criterios listados abajo,
       iterar con `skill-creator` (modo "edit/optimize") hasta que sí.
    4. Ejecutar el skill en un caso de prueba antes de darlo por
       terminado.
    5. Hacer commit del skill con `feat(skills): añade skill <nombre>`.

  PROHIBIDO escribir manualmente el `SKILL.md` de un skill sin pasar por
  `skill-creator`. PROHIBIDO inventar la estructura del archivo.

SKILL 1 — `crear-modulo`
  Propósito: scaffolding completo de un nuevo módulo de feature siguiendo
  la estructura api/domain/data/presentation, build.gradle.kts por
  convención plugin, paquetes vacíos correctos, Hilt module stub, README
  del módulo, y stubs de test en cada submódulo. Incluye stubs de
  ErrorMapper y registro de DomainError específicos del módulo.
  Inputs: nombre del módulo, tipo (feature o core), dependencias iniciales.
  Outputs: árbol de archivos creado y registro en `settings.gradle.kts`.

SKILL 2 — `crear-vista`
  Propósito: generar una pantalla Compose con:
    · Composable puro `XxxScreen(state, onEvent)` SIN ViewModel.
    · Composable wrapper `XxxRoute(viewModel = hiltViewModel())`.
    · `XxxUiState` (sealed interface incluyendo `Error(uiError)`).
    · `XxxUiEvent` (sealed interface incluyendo `Retry`).
    · `XxxUiEffect` (incluye `ShowSnackbar(uiError)` cuando aplique).
    · `@Preview` por cada estado (Loading, Empty, Error, Content) en
      claro y oscuro.
    · Uso obligatorio de componentes de `:core:design-system`
      (`MangoErrorState`, `MangoLoadingIndicator`, etc.).
    · Stub de test Compose y snapshot.
  Validar antes de escribir: si el Composable de UI importa `hiltViewModel`,
  `viewModel`, `DomainError` o `Throwable`, RECHAZAR y corregir.

SKILL 3 — `validar-arquitectura`
  Propósito: auditar el repositorio y reportar violaciones:
    · Composables de feature importando ViewModel directamente.
    · Capa data exponiendo DTOs/Entities a presentation.
    · Comunicación entre módulos sin pasar por `:api`.
    · UseCases con lógica de UI o de framework Android.
    · Imports prohibidos (Material3 fuera del design-system).
    · Reglas de dependencia entre módulos.
  Outputs: informe en español con lista de violaciones, archivo:línea, y
  sugerencia de corrección.
  Ejecutar tras cada feature.

SKILL 4 — `crear-pruebas-unitarias`
  Propósito: para un módulo dado, generar:
    · Tests de UseCases (cubrir happy path + cada `DomainError` posible).
    · Tests de Repository con MockWebServer e InMemoryRoom (incluye casos
      de timeout, 4xx, 5xx, JSON inválido, sin red).
    · Tests de Mappers (DTO↔Domain, Entity↔Domain, Throwable→DomainError,
      DomainError→UiError).
    · Tests de ViewModel con Turbine y dispatcher de test, cubriendo
      Loading/Error/Content/Retry.
    · Snapshot tests para Composables del design system (incluye estados
      de error).
  Reportar cobertura esperada y comandos para ejecutar.

SKILL 5 — `documentar-modulo`
  Propósito: por cada módulo, generar en `/<modulo>/docs/`:
    · `modulo.md` — propósito, contratos, ejemplos de uso, dependencias.
    · `diseno.md` — capas, decisiones, diagramas en mermaid.
    · `pruebas.md` — estrategia de testing y comandos.
    · `errores.md` — `DomainError` que el módulo puede emitir, mapeo a
      `UiError`, mensajes localizados.
  Lenguaje: español. Formato consistente; usar plantillas del skill.

SKILL 6 — `prompts-de-diseno`
  Propósito: para cada Composable de pantalla, generar un prompt de
  generación de imagen (compatible con generadores tipo Midjourney,
  Stable Diffusion, DALL·E) que describa:
    · Estilo Mango (paleta, tipografía, composición).
    · Layout exacto de la pantalla en su estado de contenido.
    · Variantes solicitadas (claro/oscuro, móvil, tablet).
  Outputs: archivos en `/docs/diseno/prompts/<pantalla>.md`, uno por
  estado, todos en español.

SKILL 7 — `validar-manejo-errores`
  Propósito: auditar el repositorio y reportar:
    · Repositorios sin `safeApiCall` / `safeDbCall` (o equivalente).
    · UseCases que devuelven tipos crudos en vez de `Either<DomainError, T>`.
    · Composables que muestran `throwable.message` o reciben `DomainError`.
    · `try/catch` genérico fuera de las barreras permitidas (mappers,
      CoroutineExceptionHandler raíz, init de SDK).
    · Strings de error hardcoded (deben usar `stringResource`).
    · UseCases sin tests por cada rama de `DomainError`.
    · `runCatching` sin `.fold` a `Either`.
    · Falta de `CoroutineExceptionHandler` en algún `viewModelScope`
      crítico.
    · Mappers `Throwable → DomainError` o `DomainError → UiError` sin
      tests para cada rama.
  Outputs: informe en español con archivo:línea y corrección sugerida.
  Ejecutar tras cada PR de feature.

Todos los skills se invocan vía la herramienta Skill de Claude Code y se
usan obligatoriamente durante el desarrollo. Antes de cualquier código
de feature, ejecutar los skills correspondientes.
